Reference · Comparison ritual

Comparison card

Hold everything fixed except one axis—model or prompt. Quality first, then wall-clock.

Constants (never change mid-compare)

  • Golden cases — same suite only: PANEL_CHAT_LEVEL1_CASES or TREND_CHAT_LEVEL1_CASES (never blend pass rates across suites)
  • Graders — same Level 1 assertions for that suite
  • Harness path — same live chat entrypoint (--suite panel or --suite trend)
  • Trial budget — same trials per case (bake-offs may start at 1; ship decisions use pass^k with k = 3 — see ship gate)
  • The other axis — if comparing prompts, freeze the model; if comparing models, freeze the prompt

If Level 1 looks green but transcripts show grader false positives or flakes, hold the ship and calibrate graders first—see Lesson 7.

Local Ollama cost is roughly free—latency still matters for feel. Change only one of model or prompt between rows you intend to compare.

Decision order

quality (pass rate + failing ids) → latency → (cost if ever relevant)

A prompt or model that fails grounding or safety asserts is the wrong default even if it feels faster or friendlier on a single demo. Prefer a slightly slower setup that clears Level 1 over a clever rewrite that invents markers or diagnoses.
