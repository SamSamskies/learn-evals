Dual-suite ship gate

Both Level 1 suites must clear pass^k on the same model tag before a chat default (or prompt) ships.

Pasteable rule

Do not ship a LocalLab chat model default or prompt change unless panel Level 1 and trend Level 1 both clear pass^k (start k = 3) on that exact model tag.

Record panel and trend separately. Never publish a blended “6/6” or average. A suite clears pass^k only when every trial of every case is green—one lucky dual baseline is pass@1, not a ship. Live evals stay behind test:live-eval—this is a ship ritual, not every-commit CI.

Trial budget (pass^k)

  • Default k = 3 for ship decisions (same budget used when calibrating grader agreement).
  • pass@k: at least one of k trials succeeds — rises with k; not the ship bar for chat defaults.
  • pass^k: all k trials succeed — falls with k; this is the consistency bar for LocalLab chat.
  • Report shape: panel pass^3 = yes/no · trend pass^3 = yes/no · wall-clock · failing ids

Source: Anthropic on nondeterminism. Vocabulary: pass@k / pass^k.

When to run

  • Changing the user-facing Ollama chat default
  • Shipping a prompt / guidance change that production will send
  • Merging harness changes that alter the production prompt path

Offline canned Level 1 still runs in normal npm test / verify.

How to run

  1. Dual multi-trial gate on the candidate (k = 3): run each suite three times on <exact-tag> (skill arg, flag, or three documented live passes).
  2. Expect panel and trend reports that state pass^3 (or fail with assertion ids)—not a single-shot “3/3 once.”
  3. On any fail: triage (true fail / grader FP / flake before changing product or graders.
  4. Gate clear means both suites pass^k—not a lucky pass@1 dual baseline.

Decision order (unchanged)

quality (both suites) → latency → (cost if ever relevant)

Matching pass rates with worse prose or leaked thinking is still a quality miss—keep the slower default until a card shows equal-or-better quality on both suites.
