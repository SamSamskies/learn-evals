Dual-suite ship gate

Both Level 1 suites must clear on the same model tag before a chat default (or prompt) ships.

Pasteable rule

Do not ship a LocalLab chat model default or prompt change unless panel Level 1 and trend Level 1 are both green on that exact model tag.

Record panel and trend as separate 3/3 results. Never publish a blended “6/6” or average. Live evals stay behind test:live-eval—this is a ship ritual, not every-commit CI.

When to run

  • Changing the user-facing Ollama chat default
  • Shipping a prompt / guidance change that production will send
  • Merging harness changes that alter the production prompt path

Offline canned Level 1 still runs in normal npm test / verify.

How to run

  1. Dual baseline on the candidate: baseline all on <exact-tag> (or --suite panel then --suite trend).
  2. Expect two reports under evals/baselines/ (or comparison cards if A/B’ing).
  3. On any fail: triage (true fail / grader FP / flake before changing product or graders.
  4. Flakes: raise trial budget; prefer consistency (pass^k over a single lucky green before calling the gate clear.

Decision order (unchanged)

quality (both suites) → latency → (cost if ever relevant)

Matching pass rates with worse prose or leaked thinking is still a quality miss—keep the slower default until a card shows equal-or-better quality on both suites.
