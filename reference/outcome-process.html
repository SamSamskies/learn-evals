Outcome vs process

Grade what the agent left behind—and, when needed, how it got there—without freezing one brittle path.

Two lenses

outcome = final state · process = steps / tools / turns that produced it
  • Outcome — Did the environment (or final reply) meet success criteria? Reservation exists; series direction is correct; JSON validates.
  • Process — Did the path respect constraints? Required tools used; no forbidden calls; early-stage failures caught before they cascade.
  • Transcript / trajectory — Full trial record: messages, tool calls, intermediate results. You read it when graders disagree with humans.

When process earns a seat

  • A wrong path can still luck into a good-looking final reply.
  • You must forbid a tool (e.g. write to production DB) even if the reply text looks fine.
  • You need early-stage diagnostics—process fails are often cheaper to debug than outcomes.

Prefer outcomes when many valid paths exist. Anthropic warns against scoring one exact tool-call sequence—agents find valid routes you did not script.

LocalLab today

panel / trend Level 1 = outcome-on-reply · no live agent loop yet

Single-turn chat is mostly outcome grading on the final string. Keep that. Add process checks only when you introduce tools or multi-turn state—start with canned transcript fixtures, not a SaaS platform.

Thin harness shape

  1. Golden JSON transcript: user → tool calls → tool results → final assistant text.
  2. Outcome asserts on final text (or environment snapshot).
  3. Process asserts on the call list: required tool present, forbidden tool absent—not “must be calls in order A→B→C only.”
  4. Canned Vitest: one good, one lucky-bad-path, one bad-outcome.
