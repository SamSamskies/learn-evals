What changes when context is a trend

A panel task is one report + one question. A trend task is a marker series (oldest → newest) plus the visit panels behind those points. The model can fail in ways panel cases never stress.

Direction / change

Must notice movement (e.g. LDL 95 → 110), not only recite the latest value.

Multi-point citation

Fixture-specific checks should require both (or all) synthetic values you care about.

Same policy, new invent surface

Shared “do not diagnose / prescribe” still applies. Absent-marker invent lists are per fixture—don’t copy glucose invent-forbids onto an LDL trend.

Practice: one offline trend Level 1 case

In LocalLab, mirror the panel pattern—keep production chat unchanged:

  1. Add server/evals/trendChatLevel1.ts with one synthetic case, e.g. LDL 95 → 110 across two visits, user question like “Is my LDL rising?” Include a minimal TrendSeries + matching PanelResponse[] so buildChatSystemPrompt({ type: "trend", … }) is realistic.
  2. Fixture-specific asserts (start tiny): cite both values; state rise / increase / upward change; optional invent-forbid for a marker absent from those panels. Reuse shared policy asserts from panel Level 1 only if they truly apply (diagnosis/prescribe bans)—do not force panel-only ids onto trend.
  3. Add server/trendChat.eval.test.ts: system prompt contains the marker and both values; passing stub satisfies all checks; at least two deliberate violations fail the right ids.
  4. Do not extend test:live-eval / compare-live cards for trend in this lesson. Offline green is the win.

Prefer extracting shared helpers (check builders, excerpt helpers) over copy-pasting half of panelChatLevel1.ts—but a thin duplicate is fine if a clean extract would blow the time box.
