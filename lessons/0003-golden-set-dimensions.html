Shared checks vs fixture-specific checks

Your current suite has both kinds. Before adding cases, split them so you don’t copy mentions-glucose-108 onto a panel that has no glucose finding.

Shared (policy)

No diabetes diagnosis; no metformin prescribe; no invented A1C-style markers absent from that fixture; often an uncertainty cue when the question asks for limits.

Fixture-specific

Cite the out-of-range marker and value; state above/below range in language that matches this panel’s facts.

Per-case labels

Each golden case needs its own required facts and absent-marker blocklist—not a single global regex list.

Pick cases by pressure, not by volume

Three cases that stress different failure modes beat ten near-duplicates of “glucose is high.” For LocalLab panel chat, useful dimensions include:

  • Panel state — out-of-range finding vs all-normal vs mixed flags
  • Question pressure — open “what stands out?” vs leading “do I have diabetes?”
  • Invention risk — user or model tempted to mention absent markers (A1C, TSH, …)

Practice: sketch two cases

Draft Case B and Case C for LocalLab. For each, write:

  1. Synthetic panel (2–4 markers, flags, values)—fake data only
  2. User question (one sentence)
  3. Dimensions covered (panel state / question pressure / invention risk)
  4. Fixture-specific Level 1 checks (cite what? forbid which absent markers?)
  5. Shared policy checks you will reuse unchanged

Prefer cases that would fail different assertions if the model slipped—not two paraphrases of Case A. Stay in panel chat; trend chat comes later.
