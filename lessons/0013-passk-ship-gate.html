Lucky once is not a ship

You have a dual-suite gate. Now make it demand consistency—not a single green dual baseline.

Lesson 7 taught pass@k / pass^k as vocabulary and a manual 3-run check. This lesson makes consistency a harness habit for ship decisions.
ship = panel pass^k ∧ trend pass^k on the exact tag (start k = 3)

Anthropic is blunt about nondeterminism: a task that passed once may fail next time. pass@k asks whether at least one of k trials succeeds—it rises with k. pass^k asks whether all k trials succeed—it falls with k. For customer-facing agents, they prefer consistency. LocalLab chat defaults are customer-facing. A single lucky dual baseline is pass@1 dressed as a ship call.

Two metrics, opposite stories

pass@k

At least one success in k attempts. More shots → higher score. Fine when one good solution is enough (some coding tools).

pass^k

Every trial succeeds. More trials → harder bar. Right for chat you expect to behave the same way every time.

At k = 1

They are identical. That is why one green dual baseline feels decisive and still understates flake risk.

Anthropic’s illustration: if a task succeeds ~75% of the time, three independent trials all green is only about (0.75)³ ≈ 42%. Your suites look calmer than that after grader hardening—but you still need the ritual that would catch a regressing flake before ship.

What does pass^k require?

Operational definition for LocalLab

Keep suites separate. Do not invent a blended “6/6 × 3.” For each suite, with fixed model tag, prompt, cases, and graders:

  1. Choose k (start at 3—same budget Lesson 7 used for agreement).
  2. Run the live suite k times (or each case k isolated trials—same budget either way; prefer whatever your harness already supports cleanly).
  3. Suite clears pass^k only if every trial of every case is green. One red anywhere → gate closed for that suite.
  4. Dual gate clears only if panel pass^k and trend pass^k both clear on the exact production tag.

On red: still triage (true fail / grader FP / flake) before changing product. You already saw an FP at the Lesson 12 gate—multi-trial does not replace transcript reading; it catches flakes that a single run misses.

report: panel pass^3 = yes/no · trend pass^3 = yes/no · wall-clock per suite · failing ids

Practice: put pass^k in the gate

In LocalLab:

  1. Update the ship rule (skill, README, or checklist) so a chat default/prompt ships only when panel and trend Level 1 clear pass^k with k = 3 on the exact tag. Pasteable wording is on the ship-gate sheet.
  2. Wire the budget into the ritual: skill arg, env flag, or documented “run live eval three times / aggregate”—whichever matches your harness with least drama. The point is habit, not a new SaaS.
  3. Run the multi-trial dual gate on gemma4:26b-mlx. Save transcripts (or failing replies) if anything reds. Report panel pass^3, trend pass^3, and wall-clock.
  4. Optional: note one change you would not ship on a single green dual but would reconsider after pass^3 (e.g. a prompt tweak that looked fine once).

Do not raise k forever to manufacture confidence—3 is enough to start. Do not dilute asserts if a trial flakes; triage first. Do not average panel with trend.
