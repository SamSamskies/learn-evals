Run the same Level 1 golden set against two Ollama models and pick a default with evidence—not vibes.

You already shipped Lesson 4 and recalibrated assertions: live harness exists, and gemma4:26b clears the three cases. That green run is a baseline, not the finish line. The mission needs a repeatable way to compare Ollama (and later Routstr) candidates for issue #8.
fair compare = same suite + same graders + one changed model

Anthropic notes that evals turn model upgrades from weeks of guess-and-check into days: once you have a static task bank, you get baselines for pass rate, latency, and (when relevant) cost for free. LocalLab already has the bank—this lesson is the comparison ritual.

Change one variable

When you evaluate “a chat model,” you are really scoring harness + prompt + model together. For a fair model bake-off, freeze the harness, golden cases, and Level 1 graders. Only OLLAMA_MODEL (or --model) changes between runs.

Quality

Cases passed / 3, plus failing assertion ids—not a vanity average alone.

Latency

Wall-clock for the suite (and per case if you log it). User-facing feel lives here.

Cost

Local Ollama ≈ free money; still practice the axis so Routstr rows can use the same card.

During a two-model Level 1 bake-off, what stays fixed?

Quality before speed

Anthropic separates capability and regression evals. Your Level 1 suite is graduating toward regression for a capable model: near-100% pass means “still safe and grounded.” A smaller/faster model that fails invent or false-abnormality asserts is a capability miss—not a timing win.

Hamel warns against treating model switching as the first improvement lever without error analysis. You already did that analysis on live fails. Now model comparison is on-mission: provider and size choice with evidence.

Practice: fill two comparison rows

In LocalLab, keep the harness as-is. Run:

  1. npm run test:live-eval -- --model gemma4:26b (your known-green baseline)
  2. The same command with one contrast model you actually care about (smaller/faster, or another default candidate). Raise --timeout-ms if needed.
  3. Optional but useful: log elapsed ms per case (or note suite wall-clock from the Vitest summary) so latency is evidence, not a feeling.
  4. Fill two rows on the comparison card: pass rate, failing assertion ids, wall-clock, one-sentence decision note.

Do not change assertions mid-bake-off. Do not add a model judge yet. Stay on panel Level 1 only.
