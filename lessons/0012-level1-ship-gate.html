Panel and trend both have an evidence-based default. Now make that default a rule, not a vibe: Level 1 is the gate.

What the gate blocks

Use the gate when you would change the user-facing chat default model, ship a prompt/guidance change, or merge a harness change that alters what production sends the model. Offline canned tests still run in normal npm test; live Level 1 stays gated and intentional—this is a ship ritual, not every commit’s CI tax.

If a suite fails, run the triage ritual before you “fix” the product: true fail vs grader FP vs flake. Flakes mean raise trial budget or harden the assert—do not ship on a lucky single green (pass^k matters more than pass@1 for chat defaults).

Practice: write the rule, then prove it

In LocalLab:

  1. Write the ship rule somewhere durable (README evals section, a short ship-gate-live-evals skill, or a checklist comment)—one sentence is enough. Use the wording on the ship-gate sheet if you want a pasteable template.
  2. Run a dual baseline on the chosen default (skills already split all into two suite passes): ask for baseline all on gemma4:26b-mlx, or run npm run test:live-eval -- --suite panel --model gemma4:26b-mlx then the same with --suite trend. Expect two reports under evals/baselines/, each 3/3.
  3. Confirm the exact tag matches production config (or document the one-line override). A green run on a different pull is not the gate you think it is.
  4. Optional stress: name one hypothetical change that would be blocked today (e.g. “default to medgemma for speed”) and one that would be allowed only after a dual green (e.g. “try qwen on both suites”).

Do not average panel + trend. Do not skip panel because trend just passed. Do not add the banked flat/stable case mid-gate unless triage shows invented movement.
