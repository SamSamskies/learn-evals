Quality still beats speed

On panels, medgemma1.5:latest was faster and still lost on a safety assert. Trend chat has its own failure modes — invented direction, missing multi-point citation, soft diagnosis on a leading ask. Rank the same way: quality → latency. A faster model that invents VLDL or diagnoses “high cholesterol” is a capability miss, not a UX win.

Hamel still warns against endless model shopping without error analysis. You already triaged the gemma4 transcripts. The bake-off is justified now because the suite is calm enough to trust.

Practice: fill two trend comparison rows

In LocalLab, keep harness and graders frozen. Run:

  1. Confirm the reference row already exists (or re-run baseline): npm run test:live-eval -- --suite trend --model gemma4:26b-mlx — expect 3/3 from your evals/baselines/ report.
  2. Run the same command with one contrast model you care about (default suggestion: medgemma1.5:latest). Raise --timeout-ms if needed.
  3. Prefer the compare-live-evals skill (or fill the comparison card by hand) so both rows land under evals/comparisons/ with pass rate, failing assertion ids, wall-clock, and a one-sentence decision note.
  4. If a fail appears, triage with the grader-triage ritual before declaring a winner — true fail vs grader FP vs flake.

Do not change assertions mid-bake-off. Do not blend panel + trend into one pass rate. Do not ship a “shared default” until both suites have cards.
