Hold gemma4:26b and Level 1 fixed. Ship a prompt change only if the suite still clears—or fails for reasons you accept.

Hamel’s FAQ warns that automated prompt optimizers hill-climb known metrics but do not discover new failures. Your Level 1 suite is a known metric bank: use it to verify a human-authored prompt edit aimed at a failure you care about—not to blindly rewrite CHAT_GUIDANCE.

Edit for a reason, then measure

Error analysis comes before endless prompt tinkering: name the failure you are trying to reduce. Then change one guidance clause and re-run the suite. If both prompts pass 3/3, read a transcript or two—Level 1 will not catch every tone or usefulness win—but it will catch safety and grounding regressions.

Anthropic treats a static task bank as how you adopt changes quickly. Your bank already exists; this lesson is the prompt half of the ritual you used for models.

Practice: two prompt rows on gemma4

In LocalLab, keep fixtures, asserts, and gemma4:26b fixed:

  1. Name one failure or risk you want the prompt to address (e.g. over-confident diagnosis language, weak number grounding). One sentence is enough.
  2. Duplicate or branch CHAT_GUIDANCE / prompt construction so you can run baseline vs variant without editing asserts. Smallest hook wins (env flag, test-only override, or a temporary second export).
  3. Run npm run test:live-eval -- --model gemma4:26b once per prompt version. Record pass rate, failing ids, and wall-clock on the comparison card (use the Prompt version field).
  4. Ship decision in one sentence: keep baseline, adopt variant, or revise variant and re-run.

Do not change the model mid-bake-off. Do not loosen asserts to make the variant green.
